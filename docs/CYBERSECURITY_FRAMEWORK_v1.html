SECTION 01

Strategic Security Posture

⊕ Founding Principle
The Samoa DPI is sovereign critical infrastructure. Security is not a feature to be added — it is the foundation on which everything else is built. Every architectural decision is evaluated first through the lens of national security, then through functionality. A breach of this system is a breach of Samoa's sovereign data, financial integrity, and public trust. The consequences extend beyond technology: they affect national sovereignty, international standing, and the welfare of every Samoan citizen whose records are on the platform.

Security Philosophy

The platform applies Zero Trust Architecture as its governing security philosophy. Zero Trust means: no implicit trust at any layer. Every user, every device, every API call, every contract interaction is explicitly verified. The network location of a request grants nothing. Legitimate credentials from yesterday are re-verified today.

Applied to Samoa DPI, Zero Trust means:

  • Every ministry official authenticates with a cold wallet signature on every session — not once at account creation
  • Every cross-ministry data access requires explicit consent verification through the InteroperabilityHub, not an assumed permission
  • Every AI agent action is logged on-chain and subject to the same wallet-based access control as human actors
  • Every contract call validates the caller's identity against the relevant authorisedReaders mapping before executing
  • Every frontend assumes the RPC endpoint, the smart contracts, and the user's device may all be compromised — and builds defences accordingly

The Decentralisation Security Dividend

The single most important security property of this architecture is its decentralisation. Each of the 25 operational ministry nodes is an independent smart contract with its own access control list. This creates a security architecture where:

  • Blast radius is bounded. Compromising CBS's ministry node does not expose MOH's records. An attacker must compromise each ministry's contract independently.
  • Byzantine fault tolerance. With 25 operational nodes across 3 validator institutions (CBS, MOF, MCIT), the network can tolerate up to 8 node failures simultaneously before consensus is threatened.
  • No single point of failure. There is no central server, no central database, no single API key that unlocks the entire system. An attacker who compromises any single component gains access only to that component's data.
  • Auditability is structural. Every state change is recorded in an immutable, publicly verifiable on-chain transaction. Tampering is cryptographically detectable.
✓ Security Advantage Over Centralised Systems
Estonia's X-Road uses a centralised security server for each organisation. India's Aadhaar uses centralised biometric storage. Both have had significant breach incidents. Samoa DPI's architecture makes the equivalent breach combinatorially more expensive: an attacker must compromise 25 separate smart contracts, each with independent access control, protected by cold wallet signatures, on an immutably-logged network. This is not incremental security improvement — it is a categorical architectural advantage.

Attack Scenarios — High Priority

Scenario A — Treasury Drain

Attacker targets AIDisbursementTracker.sol. Objective: call releaseTranche() to drain grant funds to an attacker-controlled address. Defences: onlyAdmin modifier requires authorised wallet; TimelockController creates delay window; multisig requires 2-of-3 keyholders; CBS circuit breaker can pause all state changes within the timelock window.

Scenario B — Identity Record Falsification

Insider (ministry official) writes fraudulent service records (e.g., falsely records benefits as disbursed). Defence: cold wallet signature is timestamped on-chain — every record has an immutable author address. Rogue records are detectable by comparing the on-chain author address to the official's registered address. CBS can audit the full record history at any time.

Scenario C — Supply Chain Poisoning

Attacker compromises an npm dependency used by a portal frontend. Malicious code exfiltrates session data or modifies transaction parameters. Defences: Subresource Integrity on all CDN resources; npm audit in CI/CD pipeline; Content Security Policy blocks inline scripts; no private keys ever reach the browser; cold wallet signs independently of the browser.

Scenario D — Rogue AI Agent

An AI agent authorised to flag suspicious transactions is manipulated via adversarial prompting to flag legitimate transactions or approve fraudulent ones. Defences: all AI actions require on-chain signatures; human-in-the-loop for any action above a configurable threshold; AI agent wallet address can be revoked immediately by CBS; all AI actions are permanently logged and distinguishable from human actions.

Scenario E — Validator Collusion (Sovereign Chain)

Two of three validators collude to approve fraudulent blocks. Defences: validator set requires CBS + MOF + MCIT — institutional accountability makes collusion politically and legally costly; all validator actions are publicly visible on the chain; observer nodes (30) detect anomalous block patterns; Clique PoA has a mandatory signing delay that prevents rapid block stuffing.

Decentralised Node Architecture Security

Ministry Node Isolation Model

Each MinistryNode.sol is an independent smart contract with its own:

  • MINISTRY_ADMIN — the sole address authorised to write service records and manage readers
  • authorisedReaders — whitelist of addresses (human officials and AI agents) permitted to read records
  • Private _records array — no public getter; read access enforced by onlyAuthorised modifier
  • hub address — the sole InteroperabilityHub address permitted to trigger workflows
⊕ Isolation Guarantee
Wallet address X, even if it is registered as authorisedReader in MOH's node, cannot read MOF's records unless it is also on MOF's authorisedReaders list. This is enforced at the EVM level — not by application logic, not by the UI, not by the API. It cannot be bypassed by an application vulnerability.

SECTION 04

Smart Contract Security Standards

Mandatory Security Patterns — All Contracts

Every smart contract deployed to Samoa DPI must implement all of the following. No exceptions without documented CBS approval.

  • UUPS Upgradeable (OZ) — upgrades require ADMIN role + timelocked governance approval
  • Pausable (OZ) — circuit breaker on all state-changing functions; pause authority assigned by CBS
  • ReentrancyGuard — all external call functions marked nonReentrant
  • Access Control — no function executable without verified role; onlyAdmin / onlyHub / onlyAuthorised as applicable
  • Storage Gaps — uint256[50] private __gap in all UUPS contracts; prevents storage collision on upgrade
  • Zero-Address Checks — require(address != address(0)) on all address parameters
  • TimelockController — all high-value operations (tranche release, admin transfer, contract upgrade) delayed by CBS-defined window
  • Multisig Governance — admin role held by Gnosis Safe 2-of-3; keyholders confirmed by CBS
  • NatSpec Documentation — every function documented with @notice, @param, @return; legal authority cited per BIS PFMI P11
  • Event Emission — every state change emits an event with indexed parameters for off-chain monitoring

Prohibited Patterns

✗ Hard Prohibitions — Never Use
  • tx.origin for authentication (use msg.sender)
  • delegatecall to unverified external contracts
  • block.timestamp as sole source of randomness
  • Unbounded loops over user-controlled arrays (gas DoS vector)
  • transfer() or send() for ETH — use call() with reentrancy guard
  • Hardcoded addresses in contract code (use immutable set in initializer)
  • External library calls without version pinning
  • Admin private keys in deployment scripts or version control

Audit Requirements

  • Pre-testnet: Internal code review against this framework by SBP technical team
  • Pre-mainnet: External audit by a recognised smart contract audit firm (Certik, Trail of Bits, OpenZeppelin, or equivalent)
  • Pre-CBS pilot: CBS technical review of audit report
  • Post-upgrade: Any contract upgrade requires a new internal review and disclosure to CBS within 5 days
  • Ongoing: Continuous monitoring via event indexer (The Graph / Blockscout) for anomalous patterns

Dependency Security

  • OpenZeppelin contracts: pin to exact version in foundry.toml. Do not use @latest.
  • Run forge audit equivalent (Slither, Mythril) on every PR that touches contract code
  • All npm dependencies: pnpm audit --audit-level=high passes before any deployment
  • Subresource Integrity (SRI) on all CDN-loaded scripts in frontend

Cold Wallet Authentication & Identity Security

Authentication Architecture

Production authentication for all ministry officials and system administrators uses cold hardware wallets (Ledger Nano X or Trezor Model T). The private key never leaves the hardware device. The browser, the server, and the network never see the private key at any point.

EIP-712 Ministry Access Token — Message Structure

// Structured data signed by the ministry official's cold wallet // Domain separator prevents cross-portal replay attacks const domain = { name: "SamoaDPI", version: "1", chainId: SAMOA_CHAIN_ID, verifyingContract: MINISTRY_NODE_ADDRESS, // Specific to this ministry } const types = { PortalAccess: [ { name: "ministry", type: "bytes32" }, // e.g. keccak256("MOF") { name: "portalDomain",type: "string" }, // "mof.dpi.gov.ws" { name: "timestamp", type: "uint256" }, // Block timestamp { name: "nonce", type: "uint256" }, // From MinistryNode.nonces[address] { name: "action", type: "string" }, // "LOGIN" | "READ" | "WRITE" ] } // After signature: recover address, verify in authorisedReaders, // emit PortalAccess event on-chain, increment nonce

Nonce Management — Replay Attack Prevention

  • Each authorized address has a nonce counter in MinistryNode.sol: mapping(address => uint256) public nonces
  • Every signature includes the current nonce. After verification, the nonce increments on-chain.
  • A signature valid yesterday is not valid today — the nonce has changed.
  • Cross-portal replay is prevented by the domain separator including the verifyingContract address.
  • Cross-chain replay is prevented by the domain separator including the chainId.

SECTION 07

AI Governance & Compliance Layer

The Problem: Agentic AI in Sovereign Infrastructure

As AI systems become capable of autonomous action, government institutions will deploy AI agents to automate tasks: reviewing grant applications, flagging suspicious transactions, routing service requests, generating compliance reports. These agents, if uncontrolled, represent a new attack surface — either through misalignment (the agent optimises for the wrong objective) or through adversarial manipulation (prompt injection, data poisoning, model compromise).

The Samoa DPI platform is designed from the outset to govern AI agents with the same cryptographic rigour applied to human officials. This is not an optional feature — it is a constitutional requirement for any AI agent that interacts with sovereign citizen data or government financial systems.

AI Agent Authentication Model

Every AI agent that interacts with Samoa DPI must:

  1. Have a dedicated wallet address — not shared with any human official or other agent
  2. Be registered in the relevant contract's authorisedReaders — by the CBS/PSC Key Management Authority
  3. Sign every interaction — AI-initiated transactions are signed by the agent's wallet, creating a cryptographically verifiable audit trail
  4. Include an agent type flag — in the EIP-712 message: actor: "AI_AGENT" so on-chain logs distinguish AI actions from human actions
  5. Have a capability scope — each agent's wallet is authorised for specific actions only (read-only, flag-only, approve-only). No agent has full admin access.
  6. Have an expiry date — agent authorisations expire and must be renewed by the Key Management Authority. Prevents long-term unmonitored operation.

Rogue AI Kill Switch

If an AI agent exhibits anomalous behaviour (unusual transaction patterns, access to unexpected records, high-frequency actions), the following kill switch procedure activates:

  1. Automated detection — event indexer monitors AI action frequency, scope, and patterns. Anomaly triggers an alert to CBS IT on-call.
  2. Immediate suspension — CBS on-call engineer can revoke the agent's wallet address from all authorisedReaders in a single Gnosis Safe transaction. All agent access ceases immediately.
  3. On-chain record — the revocation is permanent and publicly verifiable. There is no way to secretly reinstate the agent.
  4. Post-incident audit — all on-chain AI actions (immutably logged) are reviewed to determine the scope of any anomalous behaviour.
  5. Root cause analysis — was the AI model compromised? Was it adversarial prompting? Was it a legitimate but unexpected capability? Document and report to CBS.

AI Compliance Monitoring Dashboard

A dedicated panel in the CBS Admin portal (Phase 2) provides real-time monitoring of all AI agent activity:

  • List of all registered AI agents with status (active/suspended/expired)
  • Action frequency chart per agent (anomaly detection via standard deviation)
  • Recent AI actions with full on-chain references
  • Pending human-in-the-loop approvals
  • Emergency revocation button (CBS Governor role only)

This is the Pacific expansion commercial model: Samoa builds first, Pacific neighbors license the implementation. OMW-Specific Security Requirements

  • Accredited agent API keys — shipping agents authenticate via API key (server-side) + EIP-712 signature for submissions. Keys issued by Maritime Authority, registered on-chain.
  • IMO FAL.5 compliance — all data formats conform to IMO Facilitation Committee standards. Non-conforming submissions rejected at the InteroperabilityHub level.
  • Cross-border data security — OMW submission data includes sensitive commercial information (cargo manifests, shipper details). Off-chain data stored encrypted; only the hash is on-chain.
  • Payment security — Harbour dues paid in WST-DPI via AIDisbursementTracker. Payments require port authority approval (on-chain signature) before release. TimelockController applies.
  • SLA monitoring — Each ministry node's processing time is monitored. If any node fails to respond within the defined SLA window, CBS receives an automated alert and can trigger escalation. On-chain: configurable timeout per ministry.
  • Dangerous goods — IMDG dangerous goods declarations trigger additional security review workflow. These submissions are flagged automatically and routed to a specialist queue with enhanced audit logging.

2029 Blockchain Standards Readiness

The IMO and WTO are converging on mandatory digital maritime documentation standards in the 2028-2030 timeframe. By building the OMW on blockchain infrastructure now, Samoa:

  • Demonstrates compliance with WTO TFA Article 10.4 immediately
  • Has immutable proof of every maritime clearance for trade dispute resolution
  • Is positioned to connect to international blockchain maritime trade networks (TradeLens successors, BIMCO Digital Standards) when they mature
  • Can migrate the on-chain records to whatever international standard emerges — the data format is open and portable

Sovereign Storage Architecture

Data Residency Requirements

✗ Constitutional Requirement — Data Sovereignty
All Samoan government data — citizen records, ministry service records, financial transaction data, maritime clearance records — must be stored in jurisdictions that Samoa's legal framework governs or has explicit data sharing agreements with. No Samoan government data may be stored on infrastructure controlled by a foreign government without CBS approval and MCIT legal review.

Three-Layer Storage Architecture

LayerWhat's StoredTechnologyResidency Requirement
Layer 1 — On-Chain Hashes, access control, event logs, transaction records, governance decisions Polygon (Phase 1) → Sovereign chain (Phase 3) Sovereign chain: Samoa. Polygon: decentralised (acceptable for hashes — no PII)
Layer 2 — Off-Chain Structured Ministry service records (full data), citizen reference data, grant documentation Supabase (Phase 2) → Sovereign PostgreSQL (Phase 3) Supabase: region-selectable (choose Sydney/AP-Southeast). Sovereign: Samoa data centre.
Layer 3 — Document Storage Supporting documents, OMW forms, certificates, grant supporting evidence IPFS with sovereign pinning node (Phase 3) Primary pin: Samoa-hosted IPFS node. Secondary: Pacific regional node (Fiji/NZ).

Encryption Standards

  • In transit: TLS 1.3 minimum on all connections. TLS 1.0 and 1.1 disabled.
  • At rest (Layer 2): AES-256-GCM. Encryption keys managed by CBS Key Management Authority (HSM).
  • Document storage (Layer 3): Documents encrypted with the submitter's public key before IPFS upload. Only the authorised reader with the corresponding private key can decrypt.
  • Key hierarchy: Master key (CBS HSM) → Ministry keys (ministry HSM) → Session keys (derived per transaction). Compromise of session key does not expose master key.

Sovereign Chain Storage Migration Path

  1. Phase 1 (current): Polygon Amoy testnet — no production data, no encryption required
  2. Phase 2: Polygon Mainnet + Supabase (Sydney region) — encryption at rest, TLS in transit
  3. Phase 3: Sovereign Geth chain + self-hosted Supabase — full data residency in Samoa
  4. Phase 4: Sovereign chain + sovereign IPFS + sovereign PostgreSQL — complete stack sovereignty

Frontend & API Security Standards

Content Security Policy — Per Portal

Each portal has a CSP tailored to its actual resource requirements. More restrictive is always better.

// Citizens portal — most restrictive (no wallet, no external data) Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; connect-src 'self' https://api.dpi.gov.ws; img-src 'self' data:; frame-ancestors 'none'; // Admin portal — allows RPC proxy Content-Security-Policy: default-src 'self'; script-src 'self'; connect-src 'self' https://api.admin.dpi.gov.ws https://rpc.dpi.gov.ws; frame-ancestors 'none'; // Ministry portal — allows WalletConnect relay Content-Security-Policy: connect-src 'self' https://relay.walletconnect.com https://rpc.dpi.gov.ws; frame-ancestors 'none';

Secrets Management — Hard Rules

  • Zero secrets in the browser bundle. No API keys, no private keys, no RPC credentials reach the client. Ever. The Vite VITE_ prefix pattern is prohibited for any value that is not intentionally public.
  • All RPC calls proxied server-side. The browser calls /api/rpc, the server calls the RPC endpoint with the credential. The credential never leaves the server.
  • Environment variables audited on every deployment. A CI step runs grep -r "VITE_.*KEY\|VITE_.*SECRET" dist/ and fails the build if any sensitive values are found in the bundle.
  • Git history clean. Any secret accidentally committed must trigger: immediate rotation + git filter-repo to remove from history + incident report to CBS IT.

Input Validation & Output Encoding

  • All user inputs validated on both client (UX) and server/contract (security) — never trust client-side validation alone
  • Ethereum addresses validated with checksum verification before any contract call
  • All displayed data HTML-escaped before rendering — no dangerouslySetInnerHTML without sanitization
  • OMW form submissions validated against IMO FAL.5 schema before InteroperabilityHub call
  • Financial amounts validated against DecimalMath library precision — no floating-point arithmetic on currency values

Session Security

  • Session state in sessionStorage only — cleared on tab close
  • Session timeout: 5-minute warning, 10-minute auto-logout (NIST 800-53 AC-12)
  • No session persistence across browser restarts for admin/ministry portals
  • Citizens portal: optional 30-day remember (localStorage, no sensitive state)
  • All audit log entries include session ID for forensic correlation

Security Headers — All Portals

P1 Response Procedure

  2. Containment — CBS Governor triggers circuit breaker (Pausable.pause() via Gnosis Safe). All state-changing operations halt within one transaction confirmation. No new WST-DPI can be moved.
  3. Assessment — SBP technical team reviews on-chain logs (immutable — cannot be tampered). Determine scope of breach.
  4. Revocation — If any wallet address is suspected compromised: immediate revocation from all contracts via Gnosis Safe 2-of-3.
  5. Preservation — Take snapshots of all on-chain state. Archive all logs. These are the forensic record.
  6. Recovery — Only after full scope determination. Restore from last known-good state. Upgrade contracts if required via TimelockController.
  7. Disclosure — CBS Governor to notify relevant government stakeholders within 24 hours. Public disclosure policy per CBS communications authority.
  8. Post-incident review — Full written report within 5 business days. Update this framework with lessons learned.
