05 · Ministry Portal — Subdomain-Per-Ministry Architecture
The ministry portal is NOT a single URL with filtering. Each ministry gets its own authenticated subdomain. Data isolation is enforced at the smart contract level, not the UI level. A MOF official on mof.dpi.gov.ws sees only MOF data because the MinistryNode(MOF) contract rejects any wallet address not on its authorisedReaders list. The browser URL is the first trust signal; the contract is the enforcement layer.

Project Guide — Strategic Vision

What this platform is: Samoa's sovereign digital public infrastructure — the first Pacific Island nation to unify CBDC, blockchain identity, government interoperability, and retail distribution on a single cryptographically-verified platform.

How it beats Estonia: X-Road uses centralised PKI and centralised audit logs. Samoa DPI uses on-chain access control (immutable, decentralised) and on-chain audit events (tamper-evident). No single authority can secretly access another ministry's data.

How it beats India: Aadhaar stores biometric data centrally with significant privacy risk. NDIDS stores only cryptographic hashes — no PII on-chain. Consent is enforced by smart contract, not administrative process.

The Pacific advantage: Designed from day one for low-bandwidth mobile users, bilingual SM/EN interfaces, and the institutional capacity constraints of a SIDS. Replicable across Tonga, Fiji, Vanuatu, Solomon Islands, PNG using the same architecture.

Build order: Phase 1 (research submission) → Phase 2 (authentication + shared components) → Phase 3 (ministry subdomains + DBS + donor + verify) → Phase 4 (production .gov.ws + real auth + full institutional registry).

The new car test: Every portal must be immediately legible to a government official encountering it for the first time. If they need to read a manual, the UX has failed. Test every panel with someone who has never seen it before.
